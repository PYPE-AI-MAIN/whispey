# Pype Voice Analytics Dashboard — Changes
March 2026

---

## 1. Call Log Tagging (New Feature)

**Problem:** No way to annotate or categorise calls for follow-up.

**What was built:**
- Tags column (5th column) in the call logs table
- Inline tag editor — type new tags, pick from suggestions, remove tags
- Tags stored inside existing `transcription_metrics` JSONB — no DB migration needed
- `PATCH /api/logs/call-logs/[id]/tags` — updates tags without overwriting other data
- Filter: Tags → contains / exact match
- CSV export: tags as comma-separated string
- Tag colours deterministic per tag name (8-colour palette, inline styles to avoid Tailwind purging)

## 2. Observability Page — Slow Load (14s+)

**Problem:** `pype_voice_metrics_logs` query was taking 14+ seconds.

**Root causes:**
- `select: "*"` was pulling a large `otel_spans` JSONB column per turn
- Two components queried the same table with different select strings so React Query couldn't deduplicate → 2 DB calls instead of 1

**Fix:**
- Shared explicit column list (`METRICS_LOGS_SELECT`) used in both components → 1 deduplicated request
- Removed `otel_spans` from the select entirely

---

## 3. Observability Page — 616 kB Spans Payload

**Problem:** `pype_voice_spans` was returning 616 kB on every page load.

**Root cause:** `events` JSONB column fetched for all 50 spans. Only needed inside the span detail drawer which already lazy-fetches on click.

**Fix:** Removed `events` from the paginated query.

---

## 4. Observability Page — Spans Fetched Before Needed

**Problem:** Span data was fetched on page load even when the user only viewed the Turns tab.

**Fix:** Spans only fetch when the user clicks the Trace or Waterfall tab.

---

## 5. Observability Page — Trace/Waterfall Tabs Not Visible

**Problem:** Tab buttons were hidden behind `sessionSpans.length > 0`. Since spans loaded lazily, length was always 0 → buttons never appeared → spans never loaded. Deadlock.

**Fix:** Buttons now show as long as `sessionTrace` exists (lightweight, always available). Span count shows `…` while loading.

---

## 6. All Projects Fetched on Every Page Load

**Problem:** The sidebar called `GET /api/projects` on every navigation, fetching all 34 projects just to compare one ID.

**Fix:** `projectId` is already in the URL. Removed the API call entirely.
```
Before: isEnhancedProject = project?.id === ENHANCED_PROJECT_ID  // after fetching 34 rows
After:  isEnhancedProject = projectId === ENHANCED_PROJECT_ID    // from URL, zero requests
```

## 7. Column Does Not Exist Error

**Problem:** `Error loading traces: column pype_voice_metrics_logs.metadata does not exist`

**Root cause:** `metadata` belongs to `pype_voice_call_logs`, not `pype_voice_metrics_logs`. It was incorrectly included in the shared column select.

**Fix:** Removed `metadata` from `METRICS_LOGS_SELECT`.

---

## 8. Missing DB Indexes (root cause of the 14s query)

**Problem:** No index on `session_id` in `pype_voice_metrics_logs` → full table scan on every observability page open. Gets slower as call volume grows.

**Fix:** Run once in Supabase SQL Editor:
```sql
CREATE INDEX IF NOT EXISTS idx_metrics_logs_session_id
  ON public.pype_voice_metrics_logs (session_id);

CREATE INDEX IF NOT EXISTS idx_metrics_logs_unix_timestamp
  ON public.pype_voice_metrics_logs (unix_timestamp);

CREATE INDEX IF NOT EXISTS idx_metrics_logs_session_id_ts
  ON public.pype_voice_metrics_logs (session_id, unix_timestamp ASC);
```
After indexing: query drops from ~14s to under 100ms.

## 9. Production Build Failing

**Problem:** `Module not found: Can't resolve '../../.source/server'` — build failed entirely.

**Root cause:** `fumadocs-mdx` was stuck at v11 in node_modules while `package.json` declared v14. The `server.ts` artefact is only generated by v14.

**Fix:**
- Updated `fumadocs-mdx` to v14.2.11
- Re-ran `npx fumadocs-mdx` to regenerate `.source/` files
- Removed unsupported `{ language: 'english' }` option from search route
